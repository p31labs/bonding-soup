## 1. The one-page summary

P31 already has every architectural piece needed for a complete vibcoding development environment. They have not been wired together as one product because each piece was shipped in service of a different CWP (mesh, hub, geodesic, command center, fleet). This CWP zooms out and fits the pieces together as **one substrate** that scales from **kid-button-click** through **operator-grade Cursor-equivalent** with **PHOS as the guide at every level**.

The substrate is built around the **K₄ tetra** as its atomic unit, then composes tetras into **4-tetra clusters** (16 vertices), then composes clusters into **64-vertex hubs**, then composes hubs into the open mesh — preserving Buckminster Fuller's synergetic geometry that the codebase already encodes.

**Everything AND Everyone** is structural, not aspirational:
- **Everything** — every existing P31 capability (Cognitive Passport, K₄ mesh, geodesic room, command center, fleet personas, MLS protocol path, donation rails, accessibility audit, no-telemetry posture) is composable into a development surface.
- **Everyone** — the same substrate serves a 6-year-old building a button-click marble run **and** the operator drafting a new Worker on a Chromebook in a parking lot, **and** a Phase-3 community contributor pairing on an MLS feature.

## 3. The five doctrines (the floor — inherited from PEER-COMP §5)

Repeated here because every new agent picking up Phase 2+ of this CWP must protect them:

1. **Operator-condition-aware AI personas.** Every persona invoked from a vibcoding surface (PHOS guide, mechanic, counsel, oracle, scribe, debrief, narrator, firmware, quick, triage) carries the operator's medical + cognitive context into its system prompt. No persona may invent a docket, prescribe medical advice, treat executive dysfunction as deficit, use submarine metaphors, or surveil children.
2. **K₄ family mesh as architectural primitive.** The vibcoding tetra IS a K₄. Four vertices, six edges, closed. Composition rules that preserve the K₄ shape at each level are the spine of "4 tetras link → cluster → hub → mesh" semantics.
3. **Cognitive Passport as portable personalization without surveillance.** A user's vibcoding environment (theme, tool palette, completion preferences, child-safety mode, persona allowlist) lives in the CogPass on their device. The hub never sees it. Sharing a tetra with another user means sharing a *passport gesture*, not a backend join.
4. **Measurable voice.** PHOS guidance copy must pass `verify:public-voice` and `verify:delta-language` at the build step. Kid-mode surfaces must additionally pass a kid-readability subset (planned: `verify:kid-voice`, Phase 2 of this CWP).
5. **Sub-medical-grade by design.** PHOS may guide, narrate, scaffold — never diagnose, prescribe, or substitute for clinical care. Same for any "wellness" features in the kid-mode surfaces.

These are the floor. New agents who try to import a peer practice (e.g. "cloud-IDE engagement features") that breaks one of them must walk away from the practice and document the conflict.

## 5. The "PiP CLI in browser" — what it is and what it is not

The operator's directive specified "cli interface picture in picture style inside the browser window". This is the central new affordance of Phase 2 of this CWP. To prevent scope drift, we name what it is and what it explicitly is *not*:

**It IS:**
- A floating, draggable, resizable container inside the hub page (or any vibcoding surface)
- That contains a real terminal (xterm.js or equivalent in-browser TTY)
- That speaks to **either** (a) the local command center on `127.0.0.1:3131` (operator on own device) **or** (b) a P31 sandbox Worker (visitor on a public hub surface; no shell access — only allowlisted P31 commands)
- And a Monaco editor pane next to (or inside) the same container, scoped to a vibcoding sandbox file tree
- And a PHOS persona pane that observes both and offers Guidance, Not Solutions™

**It is NOT:**
- A general-purpose remote shell. No `rm`, no `curl`, no `bash -c`. The whitelist is `npm`, `git`, `p31`, `node` against a sandbox dir only.
- A multi-tenant cloud IDE. P31 is family-scale; a vibcoding session is per-user-per-tetra.
- A Cursor competitor. Cursor is for the operator-grade lane. This is for **everyone else**, with an operator-grade lane for the operator that **borrows** the Cursor pattern but does not try to replicate it.
- A surveillance surface. The PHOS persona observes only what the user types in this session, in this tetra. No off-tetra context, no off-device transmission, no telemetry.

**Acceptance test for the PiP CLI:** A 10-year-old can drag it around, resize it, type `p31 fun`, see ASCII output, and close it without breaking any other surface state. An operator can drag it into the corner, type `npm run verify`, watch it run, and pivot to a different tetra without losing the buffer.

## 6. The 4-tetra cluster pattern — what links and how

This is the operator's "4 individual tetras can link up and work together" expanded into a contract.

### 6.1 Composition

- **Tetra (atomic unit):** 4 vertices = 4 named contexts. Each vertex holds: a `vibe-sandbox` file tree, a PiP CLI session, a PHOS persona memory, and a CogPass-scoped per-vertex config. Edges = 6 bidirectional state-sharing channels (each pair of vertices can swap one of: file, command output, PHOS prompt, CogPass profile snippet).
- **Cluster (4-tetra composition):** 4 tetras share one **bridge persona** (default: PHOS in coordinator role). Cross-tetra edges = `4 × C(2,1) × 4 = 16 inter-tetra channels` between tetra-roots; each channel carries an MLS group secret (Phase 3) so cross-tetra collaboration is forward-secret.
- **Hub (4-cluster composition):** 4 clusters share one **hub manifest**. Hub manifest = which clusters are "discoverable", what their public capability surface is (e.g. "this cluster does kid mode + spanish + calligraphy"), and how to request a join.
- **Mesh (open):** Hubs publish federation records via the planetary onboard surface. No central coordinator; discovery via known-hub gossip.

### 6.2 Why K₄ at every level

Because the operator explicitly said "this is our craft" and the K₄ has been the structural primitive of the substrate since `CLAUDE.md` §2. Every level preserves the property that **every member is one edge from every other member** — no hierarchy, no broker. At cluster level, the property holds for tetra-roots (4 nodes, complete graph). At hub level, the property holds for cluster-roots (same shape).

This is also Buckminster Fuller's **vector equilibrium** under another name. See `docs/SIC-POVM-K4-ARCHITECTURE.md` and `docs/SIC-POVM-MATHEMATICAL-APPENDIX.md` for the geometric defense.

### 6.3 What it is NOT

Not a blockchain. Not a federation protocol invention. Not a new identity system. The vibcoding cluster is **shaped** like K₄; the **transport** is plain WebSockets to a Cloudflare DO during prototype, MLS-bearing once Phase 4 ships.

## 7. The kid-button-click → operator-grade spectrum

PHOS guides the user along a spectrum. Each step is a real surface; transitions are non-coercive (CogPass remembers the user's level but never forces a change).

| Level | Surface | What user does | PHOS role | Tools enabled |
|---|---|---|---|---|
| **0 — Witness** | `/vibe?lvl=0` (planned) | Reads. Watches a build run. | Narrates ("we're now compiling"). | None. Read-only. |
| **1 — Button** | `/vibe?lvl=1` | Click colored buttons. Each button = one thing. | "The blue button makes a sound." | Big buttons. Sound. Color. No keyboard. |
| **2 — Glue** | `/vibe?lvl=2` | Drag block-like atoms together. Geodesic-room-style. | "These two atoms snap when you bring them close." | Geodesic shapes as code-blocks. Snap-to-grid. |
| **3 — Type a word** | `/vibe?lvl=3` | Type one word at a time into a slot. PHOS suggests. | "Try `red`. It will change the color." | Tiny code box, single line. Picker UI for valid tokens. |
| **4 — Full sentence** | `/vibe?lvl=4` | Write a full statement. Run. See output. | "I think this line might confuse the computer because…" | Monaco editor in-line (no PiP yet). Single file. |
| **5 — Project** | `/vibe?lvl=5` | Multi-file project. Real npm scripts. | Suggests when stuck; doesn't auto-write code. | PiP CLI. Monaco. File tree. |
| **6 — Tetra** | `/vibe?lvl=6` | Operate inside a 4-vertex tetra. Pair / swap context. | Coordinates between vertices when asked. | Everything in 5 + tetra management. |
| **7 — Cluster** | `/vibe?lvl=7` | Operate across 4 tetras. Lead a pod. | Cluster-wide coordination + briefing. | Everything in 6 + cluster nav. |
| **8 — Operator** | `/vibe?lvl=8` | Full Cursor-equivalent on the operator's own machine. | Mechanic / counsel / scribe / oracle as appropriate to the task. | Everything. Direct shell access (whitelisted). MCP bridge. Full ship-bar. |

CogPass `screenComfort` cascade + `audienceProfile` field decide which level a fresh device boots into. Operator's CogPass = level 8. A child's CogPass set by the operator = level 1 or 2.

## 10. Out of scope (explicit, not afterthought)

- Building a code-editor competitor to VS Code / Cursor / Replit. We host an editor *because* PHOS needs a buffer to observe; we do not aspire to compete with editors as products.
- Cloud-IDE multi-tenancy with a billing layer. P31's monetary contract (`andromeda/04_SOFTWARE/p31ca/ground-truth/creator-economy.json`) is platform-fee 0; cluster hosting is operator-self-hosted by default.
- Phone-number-based identity. Same as everywhere else in P31.
- Engagement features (badges, streaks, leaderboards, "wow" loops, daily challenges, push notifications). Banned by `docs/ETHICAL-STYLE-MAP.md` and `docs/PUBLIC-VOICE.md`.
- Any "kids learn to code with AI" surface that auto-writes the kid's code. PHOS guides; the kid types. The whole point is the kid types.
- A "general-purpose remote shell" capability. The PiP CLI runs an allowlisted command set; never `bash -c`; never `curl`.
- Telemetry of any kind on visitor surfaces. `verify:no-telemetry` (PEER-2D) extends to every vibcoding surface automatically.
