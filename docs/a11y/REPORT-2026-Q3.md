## The target

**WCAG 2.2 Level AA**, with the following caveats:

1. We use **WCAG 2.2** rather than 2.1 because the operator's own AuDHD informs which 2.2 additions are most relevant (notably 2.4.11 Focus Not Obscured and 2.5.7 Dragging Movements). The existing `andromeda/04_SOFTWARE/p31ca/public/accessibility.html` page declares WCAG 2.1 partial conformance; that statement remains valid for the hub pages it covers, and will be upgraded to 2.2 as part of this Phase 1 work landing in p31ca.
2. We acknowledge **partial conformance** today. This is not a marketing failure; it is a baseline. The point of Phase 1 is to make the gap measurable and to commit to closing the P0 / P1 findings.
3. We do **not** yet meet 1.4.3 Contrast Minimum across all surfaces because the static checker cannot compute color-contrast ratios. That is in scope for the in-browser audit (Phase 2 of the same CWP, when axe-core runs against rendered pages).

## What the static checker covers

`scripts/a11y-audit.mjs` (PEER-1G) implements the following checks. The list is intentionally narrow — these are the high-value, browser-free checks. The in-browser audit (Phase 2) covers the rest.

| Rule ID | WCAG | Severity | What it checks |
|---------|------|----------|----------------|
| `html-lang` | 3.1.1 (A) | error | `<html lang="…">` is set |
| `doctype` | n/a | warn | `<!DOCTYPE html>` is declared |
| `title-element` | 2.4.2 (A) | error | `<title>` exists and is non-empty |
| `viewport-meta` | 1.4.4 / 1.4.10 (AA) | warn | viewport meta is present and does not block user zoom |
| `img-alt` | 1.1.1 (A) | error | every `<img>` has an `alt` attribute (empty `alt=""` is OK for decorative) |
| `input-label` | 3.3.2 (A) | warn | every interactive `<input>` has `aria-label`, `aria-labelledby`, `title`, or a `<label for="…">` |
| `heading-h1` | 1.3.1 / 2.4.6 (A/AA) | warn | exactly one `<h1>` per page |
| `heading-skip` | 1.3.1 (A) | warn | heading levels do not skip (e.g. `<h2>` → `<h4>`) |
| `link-empty` | 2.4.4 (A) | error | every `<a>` has accessible text or `aria-label` / `title` |
| `skip-link` | 2.4.1 (A) | info | pages with `<nav>` provide a skip-to-main link |
| `main-landmark` | 1.3.1 / 2.4.1 | info | `<main>` landmark exists |
| `duplicate-id` | 4.1.1 (legacy) | warn | no duplicate `id` attributes on the same page |

The checker exits **0** by default (so it can land on the verify chain immediately without being a blocker). Run with `--strict` to fail on any error-class finding; we will flip the default to strict once the baseline is clean.

## Methodology going forward

1. **Now (Phase 1, this quarter):** static-check baseline. The verify chain runs `a11y:audit` and the operator (or a contributor) reviews the report on every PR. New error-class findings on previously-clean files block the PR.

2. **Next (Phase 2, Q4 2026):** in-browser audit. We add `npm run a11y:audit:browser` that drives Playwright + axe-core against rendered pages. This catches color contrast, ARIA tree, and computed accessible name. The report will live alongside this one at `docs/a11y/REPORT-2026-Q4.md`.

3. **Then (Phase 2, Q4 2026):** manual screen-reader walk-throughs. NVDA on Windows (primary), VoiceOver on macOS (secondary), TalkBack on Android (via cloud emulator). We document the walkthrough as a checklist in `docs/a11y/SCREEN-READER-WALKTHROUGH.md`.

4. **Public posting (continuous):** The `/accessibility` page on `p31ca.org` is the public conformance statement. It already exists for WCAG 2.1; it will be updated to 2.2 + this report's findings when the andromeda mirror PR lands.

## How findings are categorized

We adopt a four-band priority for findings:

| Priority | Definition | Triage |
|----------|------------|--------|
| **P0** | Blocks a user with a disability from completing a critical task (e.g. reading the manifesto, generating a passport, donating) | Fixed within the same quarter the finding lands |
| **P1** | Significantly impairs a user with a disability but does not block the critical task (e.g. missing skip-link on a long page, heading-skip in a documentation page) | Fixed within two quarters |
| **P2** | Standards-conformance gap with no observed impact (e.g. legacy `<font>` tags, decorative-image alt-text policy drift) | Tracked; fixed opportunistically |
| **P3** | Cosmetic / preference (e.g. `aria-describedby` on a form field that already has a label) | Welcome PRs; no commitment |

P0 and P1 findings are summarized in the next section.

## Why this matters

P31 was built around an operator with a chronic medical condition (hypoparathyroidism) and a late-diagnosed AuDHD profile. Accessibility is not a feature we are adding to be polite — it is the floor under our own house. If a P0 finding lands, the operator is one of the people it might affect. We treat it that seriously.

The static check does not replace the in-browser audit. The in-browser audit does not replace manual screen-reader testing. Manual screen-reader testing does not replace the lived experience of users with disabilities reporting back. All four are part of the picture.

---

## How to report a barrier

If you encounter an accessibility barrier on any P31 surface, see `docs/security/REPORTING.md` (we use the same channel, with the subject line `[a11y]` instead of `[security]`). Or open an issue on `github.com/p31labs/bonding-soup` with the `a11y` label.

We commit to acknowledging within 72 hours. We commit to a triage decision within 7 days. P0 findings get a same-quarter fix. P1 findings get a two-quarter fix. P2 / P3 are tracked.
