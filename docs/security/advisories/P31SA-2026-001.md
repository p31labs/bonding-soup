## Description

This advisory is filed as the **first publication** under the new `P31SA-YYYY-NNN` advisory framework introduced in `docs/CWP-P31-PEER-COMP-2026-05.md` Phase 1, task PEER-1B.

It documents a routine hardening pass against four classes of theoretical concern. **No reachable defect was found** in any of the four areas. The hardening was added because the four classes are well-understood industry concerns and adding the defenses costs little.

We file this as a Low-severity advisory rather than skipping the disclosure for two reasons:

1. To exercise the new framework end-to-end and verify the publication path (commit, link, render at the GitHub blob URL).
2. To establish the practice that defense-in-depth changes — even ones with no known reachable bug — get a public record. This is the Mozilla MFSA model: even quiet weeks get an advisory cadence so that absence of advisories never means absence of vigilance.

---

## Impact

**No known impact.** The four areas hardened (rate limiting on the donate API, input validation on the passkey worker, header-stripping on the glass-box probe responses, and a CSP refresh on the static hub) had no documented incident, no reproduction, and no observed exploitation. The hardening reduces the surface area available to a hypothetical attacker who finds a defect we have not yet found.

## Reproduction

**Not applicable.** No reproducible defect.

## Mitigations (for users on affected versions)

Upgrade to the fixed commit. There are no user-side mitigations because there was no observed exploit.

## Fix

- Commit `9d16fee` — "Ecosystem hardening + polish sweep — 4 new defenses, 0 vulns, GREEN" (the merge commit that landed the four defense-in-depth changes)

The four changes are visible in the commit's diff. They span:

1. Donate-API Worker — rate-limit envelope refresh
2. Passkey Worker — input-validation hardening on the register-begin / auth-begin endpoints
3. Glass-box probe responses — header-stripping pass to ensure no operator metadata leaks via response headers
4. Static hub — Content Security Policy refresh on the BaseLayout

Each change has its own test and is covered by `npm run security:check` (in the `andromeda/04_SOFTWARE/p31ca` workspace) and the home `verify:public-sanitization` gate.
